# Parakeet TDT 0.6B v3 — onnx-asr vs Stelnet (Apple M1, Metal)

Wider-matrix follow-up to PERFORMANCE.md § "onnx-asr cross-comparison
— issue #81 (2026-05-09)". The headline conclusion is **already in
PERFORMANCE.md** (stelnet Q8_0 Metal 1.34 s / 8.24× RT vs onnx-asr
TDT CPU EP 1.77 s / 6.23× RT — stelnet Metal wins by 1.32× on
TDT-vs-TDT and 1.58× on CTC-vs-CTC); this run sweeps a 5-quant ×
2-mode × 2-audio matrix to compare quants and streaming windows.

**Caveat on the ONNX numbers below.** This particular run was taken
with CoreMLExecutionProvider as the default ONNX EP, which is the
*wrong* default for parakeet on M1 (the CoreML EP is documented in
PERFORMANCE.md to be slower than CPU EP for parakeet-shaped graphs;
e.g. CTC int8 was 1.28 s CoreML vs 0.72 s CPU). The script's default
has since been changed to CPU EP on hosts without a working dGPU EP,
so a re-run will produce *better* ONNX numbers than the table below.
The "engine wins" framing is unchanged — the May 9 PERFORMANCE.md
already established stelnet Metal beats onnx-asr CPU EP, the EP that
was being penalized is the slower one — but the absolute ONNX
numbers here are pessimistic by ~1.7×. Re-run after the default fix
for a truer ONNX baseline.

Generated by `tools/benchmark_asr_engines.py` on 2026-05-10 against the
same `parakeet-tdt-0.6b-v3` model in two formats:

- **GGUF** via Stelnet (`stelnet_parakeet_*` C ABI through ctypes).
  Backend: Metal. Threads: 4. Flash attention: on.
- **ONNX** via [onnx-asr](https://github.com/istupakov/onnx-asr) 0.11.0.
  Provider for *this run*: `CoreMLExecutionProvider` (with
  `CPUExecutionProvider` fallback) — wrong default; see caveat above.
  For the fp32 cell the script auto-falls-back to CPU EP because the
  onnxruntime CoreML EP can't load fp32 ONNX with external-data
  sidecars: the external-data initializer + CoreML's subgraph split
  lose `model_path`; inlining hits protobuf's 2 GB ceiling. Tracked
  upstream as
  [`microsoft/onnxruntime#26355`](https://github.com/microsoft/onnxruntime/issues/26355)
  (closed *not planned*); the workaround is to pre-merge into a single
  fp32 file via `onnx.save(model, "encoder-model.onnx",
  save_as_external_data=False)`. The script doesn't do this
  automatically (it's a 2.4 GB one-time rewrite); the int8 cell ships
  as a single file and runs on CoreML without the workaround.

Audio:

- `short`: `samples/jfk.wav` — 11 s English, real audio, WER reported.
- `long`: `tests/fixtures/bench_long_60s.wav` — 60 s tiled JFK
  (deterministic, dependency-free; tiled content makes WER
  uninformative on this clip, so the script suppresses it).

Window: 4 s. Warmups: 1. Runs: 3.

Host: Apple M1, macOS 25.2, Python 3.11.8, onnxruntime CoreMLExecutionProvider.

¹ ONNX fp32 auto-fell-back to CPU EP — `onnxruntime#26355` blocks
external-data ONNX on CoreML; pre-merge into a single file (see
header notes) to run fp32 on CoreML. Note the int8 cell runs on
CoreML successfully but is *slower than CPU EP would be* for
parakeet on M1 (per PERFORMANCE.md), so the int8 numbers in the
table are also pessimistic for ONNX.

**Q8_0 GGUF on Metal is the fastest cell on the long clip (7.4× RT)**,
beating onnx-asr's int8 on CoreML by ~50%. (Note: against onnx-asr's
*CPU EP* — which is what onnx-asr users on macOS should actually use
per PERFORMANCE.md — the gap on this Apple M1 will be smaller. See the
caveat at the top.) The opposite of the Windows DirectML result
reported in #81 — onnx-asr wins on DirectML/CUDA, where the EP
partitioning of the parakeet encoder is much better. PERFORMANCE.md
frames the actionable takeaway as "which CUDA / Vulkan kernels in the
parakeet path are leaving perf on the table on dGPU" rather than
"parakeet is slow on GPU universally."

| stelnet | f16  | long (60 s)  | 8.41 s  | 7.1× |
| stelnet | q4_k | long (60 s)  | 11.80 s | 5.1× |
| onnx-asr | int8 | long (60 s)  | 138.5 s | **0.4×** ← sub-realtime |
| stelnet | q8_0 | short (11 s) | 2.47 s  | 4.4× |
| stelnet | f16  | short (11 s) | 3.50 s  | 3.1× |
| stelnet | q4_k | short (11 s) | 2.90 s  | 3.8× |
| onnx-asr | int8 | short (11 s) | 5.88 s  | 1.9× |
| onnx-asr | fp32¹| short (11 s) | 4.29 s  | 2.6× |

The 0.4× RT on `onnx-asr int8` whole-60s is the ugliest cell — that
shape blows past CoreML's pattern-match on the encoder graph and
nearly all of it falls back to CPU. onnx-asr's `recognize()` chunks
internally, so users typically don't hit this in practice; the
chunked numbers above are the realistic deployment shape.

## What CoreML actually does with the parakeet encoder on M1

From the onnxruntime warnings emitted at session creation:

```
CoreMLExecutionProvider::GetCapability:
  number of partitions supported by CoreML: 368
  number of nodes in the graph:             3249
  number of nodes supported by CoreML:      1428  (≈44%)
```

The remaining ~56% of nodes spill to CPU. That's not a Stelnet vs
onnx-asr story — it's a CoreML EP coverage story. On
DirectML/CUDA/TensorRT, EP coverage is much higher and onnx-asr will
look very different.

## Shape-dependent scaling (encoder is ~O(N²) in audio length)

Long audio is ~5.45× longer than short. If encoder cost were linear,
mean run for whole-mode would scale ~5.45×. We see:

|       | short | long  | ratio |
|-------|-------|-------|-------|
| q4_k  | 2.90  | 11.80 | 4.07× |
| q8_0  | 2.47  | 8.59  | 3.48× |
| f16   | 3.50  | 8.41  | 2.40× |

The < 5.45× ratios mean the *constant overhead* (mel + setup) dominates
on short audio more than the encoder dominates on long audio. F16 in
particular gets 2.4× scaling because the full-precision matmul is
already saturating Metal's pipelines on the short clip — there's
slack on long. This also explains why short-audio RTx looks worse
than long-audio RTx for every Stelnet cell.

## Caveats observed during this run

- **High first-cell variance**. The very first chunked-only smoke run
  saw q4_k chunked at 11× RT (323 ms p50). The matrix run later saw
  the same cell at 3.5× (1001 ms p50). Same hardware, same code,
  different Metal pipeline cache state. The script now has a
  `--prewarm` flag that JIT-compiles every (quant × shape) once before
  any timed cell — recommended for headline numbers.
- **M1 thermal**. Sustained ML workloads throttle. Run the matrix on a
  cool machine and don't repeat back-to-back if you want consistent
  RT factors.
- **`onnx-asr fp32` chunked p95 = 3891 ms**. CPU-EP latency variance
  on chunked is high; one chunk took ~4× the median. ONNX whole-mode
  doesn't show this because it averages over all chunks per recognize.
- **WER on the short clip was 0.000** for all 5 engine cells (q4_k,
  q8_0, f16, int8, fp32) — JFK is too easy a clip; agreement is
  expected. Not informative for quality comparison.

## Reproducing

```bash
# macOS / Metal
python tools/benchmark_asr_engines.py --prewarm --json /tmp/bench.json

# Windows / CUDA (after building libstelnet with CUDA)
python tools/benchmark_asr_engines.py --prewarm \
    --stelnet-lib build-cuda/src/stelnet.dll \
    --providers CUDAExecutionProvider,CPUExecutionProvider \
    --gpu-backend cuda
```

See `tools/benchmark_asr_engines.README.md` for full options.

## Cells not measured this run

`onnx-asr fp32` on long (60 s) audio in either mode. CoreML EP wasn't
usable on fp32 + external data (see header notes), so this would have
run on CPU EP; whole-mode extrapolation suggests ~4–6 minutes per run
× 4 passes ≈ 20+ minutes per cell, roughly 40 minutes total. Skipped
for time. To get fp32-on-CoreML numbers, pre-merge the encoder file
once and rerun.
